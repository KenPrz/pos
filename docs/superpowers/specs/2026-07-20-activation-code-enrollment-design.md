# Activation-code enrollment for register terminals

**Date:** 2026-07-20 · **Status:** Approved design, pre-implementation

## Problem

Today the admin sees the raw Sanctum device token — `POST /registers/enroll` and the
back-office reissue endpoint both return it in the HTTP response, and the admin pastes
it into the register's setup screen. The long-lived credential passes through human
hands, chat messages, and clipboards. There is also no dedicated lockout experience:
a revoked token is only detected at PIN-login time and dumps the user onto the generic
paste-a-token screen; mid-session revocation is misread as staff-session expiry.

## Decision summary

- **Activation codes replace raw-token flows everywhere.** Raw device tokens never
  leave the server via the API again. (Seeder still mints tokens server-side for dev.)
- **One code per terminal (1:1), single-use, 7-day expiry.** The device token obtained
  by redeeming a code stays long-lived with no expiry, as today.
- **Reissue = full lockout.** Issuing a new code revokes the register's device tokens
  *and* every staff session bound to that register, immediately.

## Data model

New migration adds three nullable columns to `registers` (no new table — the code is
strictly 1:1 with a terminal and only the latest matters; history lives in `audit_log`):

| Column | Type | Meaning |
| --- | --- | --- |
| `activation_code_lookup` | text, unique | Keyed HMAC-SHA256(code, APP_KEY). Same pattern as `pin_lookup` in `app/Domain/Auth/Pins.php`. HMAC (not plain SHA-256) so a DB dump alone cannot brute-force the ~50-bit code space offline. |
| `activation_code_expires_at` | timestamptz | Issue time + 7 days. |
| `activation_code_redeemed_at` | timestamptz | Set exactly once at redemption; a non-null value means the code is spent. |

**Code format:** 10 characters of Crockford base32 (no `0/O/1/I`), displayed and
accepted as `XXXXX-XXXXX` (hyphen optional, input normalized to uppercase). ~50 bits
of entropy. Generated by a new `App\Domain\Auth\ActivationCodes` helper (generate,
normalize, hmac) alongside the existing `Pins` helper.

## API

### `POST /api/v1/admin/registers/{register}/activation-code` — issue / reissue

Replaces `POST /admin/registers/{register}/token` (`ReissueDeviceToken`), which is
removed. New action `IssueActivationCode` (admin group, `Permissions::REGISTER_ENROLL`
authorization as before). In one `DB::transaction` with `lockForUpdate` on the register:

1. Generate code; store HMAC lookup, `expires_at = now + 7 days`, clear `redeemed_at`.
2. Revoke all the register's own tokens (`$register->tokens()->delete()`).
3. Revoke every staff session bound to the register: delete `User`-owned
   `personal_access_tokens` rows whose abilities contain `register:{id}`.
4. Audit `admin.register.code_issue`.

Returns `201 { "data": { "activation_code": "XXXXX-XXXXX", "expires_at": ... } }` —
plaintext shown exactly once, same convention as today's token reissue. First issue
and reissue are the same endpoint.

### `POST /api/v1/registers/activate` — redeem

New public (unauthenticated) route, throttled per-IP like the PIN limiter. New action
`ActivateRegister`. Body: `{ "activation_code": "..." }`. Look up by HMAC; in a
`lockForUpdate` transaction verify: code exists, `redeemed_at` is null, `expires_at`
in the future, register `is_active`. Then stamp `redeemed_at`, mint the no-expiry
Sanctum token with ability `device` (name `device:{id}`, as today), audit
`register.activate`. Returns `201 { "data": { "device_token": ..., "register": { id,
name, mode } } }`.

Errors (added to the table in `docs/03-api.md`):

| Code | HTTP | When |
| --- | --- | --- |
| `invalid_activation_code` | 401 | Unknown **or already redeemed** — deliberately indistinguishable to avoid an enrollment oracle. |
| `activation_code_expired` | 401 | Known, unredeemed, past `expires_at`. Distinct so a legitimate installer knows to ask for a reissue rather than retyping. |

If the network dies after the server commits but before the terminal stores the token,
the code is spent and the admin reissues. No idempotency machinery for this edge —
accepted trade-off.

### `POST /api/v1/registers/enroll` — removed

The pre-M6 "create register + hand me a raw token" path. The back office's
`CreateRegister` + issue-code covers register creation; `EnrollRegister`, its
controller, request, DTOs, and route are deleted. `Permissions::REGISTER_ENROLL`
survives as the authorization for issuing codes. The seeder keeps minting device
tokens directly server-side (nothing crosses the API), so the printed dev tokens and
all three e2e scripts are untouched.

### Register-status surface

The admin register resource gains an `activation` field so the back office can render
status: `enrolled` (has a live device token), `code_pending` (+ `expires_at`),
`code_expired`, `not_enrolled`.

## Register app (`frontend/web`)

- `SetupScreen` becomes an **activation-code entry form** (same injected-callback
  pattern as `ServerSetupScreen` for testability). Submits to new
  `api.activateRegister(code)`, which stores `pos.device_token` + `pos.register_info`
  and advances to the PIN stage.
- **Disabled variant** of the same screen, shown with the banner: *"Your activation
  code has been disabled. Please contact an admin and request a new activation
  code."* — with the code form below it, exactly as specified.
- **Routing fix in `Register.tsx`:** any `ApiError` with `code ===
  'invalid_device_token'` — at PIN login *or mid-session* — clears the device token
  (`tokens.clearDevice()`) and lands on the disabled screen. Today mid-session 401s
  are misread as staff-session expiry; the branch keys on `err.code`, not just
  `status === 401`.

## Back office (`frontend/back-office`)

`RegisterEditor`'s reissue block becomes **"Issue activation code"**: same destructive
`ConfirmDialog` ("the till goes dark immediately"), code displayed once in the
elevated Card (`select-all`, "valid for 7 days, single use — copy it now, it will not
be shown again"). A `StatusPill` renders the `activation` field: enrolled / code
pending (expires …) / code expired / not enrolled. `api.registers.reissueToken`
becomes `api.registers.issueActivationCode`.

## Testing

- **Pest (backend):** redeem happy path (token really authenticates as the register);
  expired code; double redeem; unknown code; inactive register; reissue revokes device
  token AND staff sessions (old staff token 401s); reissue invalidates the previous
  unredeemed code; throttle on `/registers/activate`; audit rows for both actions;
  non-admin 403 on issue. Pattern references: `EnrollRegisterTest`,
  `LocationRegisterTest:134` (reissue), `StaffPinTest` (throttle/lockout).
- **Vitest (web):** activation screen submit/error/busy states; `Register.tsx` routes
  `invalid_device_token` (login-time and mid-session) to the disabled screen with the
  banner; api client stores token on activate.
- **Vitest (back office):** `RegisterEditor` issue flow — confirm dialog, one-time
  code display, status pill states.
- **Docs:** `docs/03-api.md` (endpoints + error table), `docs/02-data-model.md`
  (columns), CLAUDE.md/README mentions of `registers/enroll` and token pasting.

## Out of scope

- Idempotent redemption / retry-safe exchange (admin reissue covers the edge).
- Moving the device token out of `localStorage` (Tauri shell is a blind proxy; no
  Rust changes needed).
- Code delivery channels (email/print) — the admin reads the code off the screen.
