# M4 — Retail Complete Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A retail store could run a full day, including the parts that go wrong — voids, discounts, refunds with restock, external card tenders, cash movements, stock operations, and a Z-report — with the register UI restyled to the DESIGN.md console-chrome design language.

**Architecture:** Same action-class pattern as M3 (route → 3-line controller → FormRequest → Action → Resource). New domain pieces: `DiscountResolver` (Domain/Pricing — allocates order-level discounts across lines with `Money::allocateByRatios`), `StockLedger` extensions (restock/adjust/count), `ExternalCardDriver`. Frontend: a token stylesheet generated from `frontend/web/DESIGN.md` and a restyle of the register, plus void/discount/refund/card flows.

**Tech Stack:** unchanged — Laravel 13.20 / PHP 8.5, PostgreSQL 18, Pest, React 19 + TS 7 + Vite 8.

## Global Constraints

Everything from the M3 plan still binds (money integer cents via `Money`; quantities as strings validated `regex:/^\d{1,9}(\.\d{1,3})?$/`; one action = one route = one 3-line controller; actions final/no-HTTP/own their transaction; version check inside the transaction after the lock; idempotency middleware semantics; append-only financial records; snapshots; real Postgres; `declare(strict_types=1);` everywhere; validation failures are **400** `validation_failed`, domain refusals are 409/422; Eloquent never hydrates DB column defaults after `create()` — set columns explicitly; jsonb reorders keys — replay tests use `toEqual`; **record fetches must be location-scoped to the acting register** — teams scope permission checks only).

New for M4:

- **Every "money leaves" action is supervisor-gated** per `docs/05-rbac.md`: `order.line.void`, `order.discount.apply`, `order.void`, `order.reopen`, `payment.void`, `refund.create`, `shift.cash_movement`. FormRequests authorize with the exact `Permissions::` constant; tests prove cashier-denied/supervisor-allowed for each.
- **Every "money leaves" action writes an audit row** with the actor and register.
- **Voids and refunds never delete or mutate financial rows**: lines get `voided_at`/`voided_by`; payments transition `status` to `voided` (amount immutable); refunds are new rows; the original order is never edited by a refund.
- **Discount rows store resolved cents** (`order_discounts.amount_cents`); percent discounts are re-resolved and rewritten explicitly on every recalculation (never silently re-scaled — the rewrite is by code we test).
- **Stock changes only through `StockLedger`** — movement + level in the caller's transaction, every movement has a reason from the schema's check constraint.
- **Test-DB invocation:** `cd backend && DB_PORT=5433 ./vendor/bin/pest` (machine-local port remap; never edit phpunit.xml).
- **Frontend design language:** `frontend/web/DESIGN.md` is the binding spec for Task 13/14. Its hard rules: every region is a beveled plate (hard bevels, no soft shadows); warm color means action only (nav-gold nav words, amber utility, signal-orange forward/submit); red = brand/alert/destructive; sharp/chamfered corners by default, roundness only for pills/dots/circle-arrows; uppercase Arial Bold 11px +0.5px tracking chrome labels; display type outlined with hard drop shadow. Modernizations it mandates: 44px minimum touch targets, responsive collapse below ~720px. House additions: visible keyboard focus, `prefers-reduced-motion` respected, receipt print CSS stays plain black-on-white (thermal printers don't do periwinkle).

## Deliberate M4 scope cuts (do NOT build)

- **Ad-hoc discounts** (`order_discounts.discount_id = null`) → M6 back office defines them; M4 applies catalog discounts only.
- **Cashier-appliable promos** (`discounts.requires_supervisor = false`) → the cashier role has no `order.discount.apply`; auto-apply promos are an M6 concern. M4 gates all discount application at the permission.
- **Stock back-office UI** → M6 ("an admin never needs psql"). M4 ships the stock endpoints + tests only.
- **Reopen UI** → endpoint + tests only; the register flow for it arrives with tabs (M5).
- **Refunding card money** → `external_card` is explicitly not refundable through us (docs/01); refunds validate `driver: cash`.
- **Modifiers, tabs/table_ref UI, split-tender UI, order transfer** → M5.
- **Idempotency first-use race hardening, replay headers** → accepted v1 behavior (documented in roadmap M3 notes).

### Task 2: StockLedger — restock, adjust, count

**Files:**
- Modify: `backend/app/Domain/Stock/StockLedger.php`

**Interfaces:**
- `restock(string $variantId, string $locationId, Quantity $qty, string $refType, string $refId, ?string $userId): void` — positive movement, reason `'refund'` (stock coming back from a sale context; `ref_type` disambiguates `order_line` void vs `refund_line`).
- `adjust(string $variantId, string $locationId, Quantity $delta, string $reason, ?string $userId, ?string $note): void` — signed delta, reason `'adjustment'` or `'waste'` (validated by caller’s FormRequest; the ledger asserts it's one of the two). Waste deltas must be negative; plain adjustments may go either way. Never lets the level go negative: lock the level row first (like `sell`) and throw `InsufficientStock` if `level + delta < 0`.
- `countTo(string $variantId, string $locationId, Quantity $counted, ?string $userId, ?string $note): void` — locks the level, writes ONE movement with reason `'count'` and `qty_delta = counted − current` (skips the write entirely when the delta is zero — `qty_delta <> 0` is a check constraint), leaving the level exactly at `counted`.
- All go through the existing private `move()`; all assert an open transaction.

`LineAlreadyVoided` exception: code `line_already_voided`, status 409, details `['order_line_id' => ...]`. (Add to the docs error table in Task 15.)

```php
<?php
// app/Actions/Orders/VoidLine.php
declare(strict_types=1);

namespace App\Actions\Orders;

use App\Domain\Audit\AuditLogger;
use App\Domain\Money\Quantity;
use App\Domain\Pricing\OrderTotals;
use App\Domain\Stock\StockLedger;
use App\Exceptions\Domain\LineAlreadyVoided;
use App\Exceptions\Domain\OrderClosed;
use App\Exceptions\Domain\OrderVersionConflict;
use App\Models\Order;
use App\Models\OrderStatus;
use App\Models\ProductVariant;
use App\Models\Register;
use Illuminate\Support\Facades\DB;

/**
 * Lines are voided, never deleted — "what did the cashier remove, and when" is a
 * fraud question. Tracked stock goes back on the shelf in the same transaction.
 */
final class VoidLine
{
    public function __construct(
        private readonly StockLedger $stock,
        private readonly OrderTotals $totals,
        private readonly AuditLogger $audit,
    ) {}

    public function execute(VoidLineInput $in): Order
    {
        return DB::transaction(function () use ($in): Order {
            $locationId = Register::findOrFail($in->registerId)->location_id;

            $order = Order::whereKey($in->orderId)
                ->where('location_id', $locationId)
                ->lockForUpdate()
                ->firstOrFail();

            if ($order->status !== OrderStatus::Open) {
                throw new OrderClosed($order->id, $order->status->value);
            }
            if ($order->version !== $in->expectedVersion) {
                throw new OrderVersionConflict($order->id, $in->expectedVersion, $order->version);
            }

            $line = $order->lines()->whereKey($in->lineId)->firstOrFail();

            if ($line->voided_at !== null) {
                throw new LineAlreadyVoided($line->id);
            }

            $line->forceFill(['voided_at' => now(), 'voided_by' => $in->actorId])->save();

            // Restock asks about the present, not the snapshot: a variant retired since
            // the sale simply isn't restocked.
            $variant = ProductVariant::withTrashed()->find($line->variant_id);
            if ($variant !== null && $variant->track_inventory) {
                $this->stock->restock($variant->id, $order->location_id, Quantity::fromString($line->qty), 'order_line', $line->id, $in->actorId);
            }

            $this->totals->recalculate($order);
            $order->forceFill(['version' => $order->version + 1])->save();

            $this->audit->record('order.line.void', $line, $in->actorId, [
                'order_id' => $order->id, 'reason' => $in->reason, 'sku' => $line->sku_snapshot,
            ], registerId: $in->registerId);

            return $order->fresh(['lines']);
        });
    }
}
```
